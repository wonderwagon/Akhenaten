In that case, I'm stuck with having to figure it out for myself. Here's the full nerd's digest. I'll put notes and comments in a following post, which may not appear immediately, so if you wonder what 'see Note 1' is all about, be patient!

Warning - this is the result of much observation and deductive reasoning. I have observed a lot, but I won't have observed everything and my reasoning is often questionable. While I have a high degree of confidence in each individual piece of information presented, there are so many pieces that something is almost bound to be wrong.

All offsets and values are decimal

Offset

Bytes

Purpose

00 1 Status flag. Indicates whether any actions are outstanding on the building. Usually 01 but eg 06 = deleted but not unwound, 03 = just placed but not registered
01 1 Usually 01. Purpose not known
02 1 Usually 00. Purpose not known
03 1 Building size. 01 = 1x1, 02 = 2x2, etc.
04 1 Merge flag. Indicates that a house below Large Insulae is formed by merging tents
05 1 Housing: repeats data at offset 03. Non-housing set to 00
06 1 X coordinate of building's northern tile
07 1 Y coordinate of building's northern tile
08 2 Grid offset. See note 1.
10 2 Building type. This corresponds to the number shown alongside the building type in c3_model.txt
12 2? Multipurpose: Warehouse bay - goods ID. Same order as Trade Advisor 'show prices' screen, i.e 01=wheat, 15=potte Native hut - number of entry in buildings list of related meeting centre (natives spawned will walk to here then return). Gatehouse - 01 facing NE/SW 02 facing NW/SE. Workshop - 01 = oil, 02 = wine, 03 = weapons, 04 = furniture, 05 = pottery Housing - evolution level (00 = Small Tent, 01 = Large Tent, etc) Fort - 11 = Javelin, 12 = Cavalry, 13 = Infantry Hippodrome - 00 = NW section, 01 = centre section, 02 = SE section

14 2 Road network ID. Set for any building adjacent to a road (even if access is not needed) but NOT set for housing. Every separate road system (even just a single tile) has its own network ID
16 2 Consecutive number. See note 2.
18 2 Labor access points. See reply #2 of Labor Access for more details (but ignore anything I say about towers )
20 2 Labor distribution. Determines allocation of workers in the event of a labor shortage within the sector. See above thread for further info.

24 2 Housing - current vacancy. Other - not used
26 2 Distance from access tile (see offset 32) to entry point not permitting diagonal travel, ie moving along X and Y axes only. See Cartpushers, traders, etc. - which building is nearest? for details
28 2 Highest number of residents since house was placed. Food deliveries are based on this
32 1 x coordinate of 'access tile'. See above cartpushers thread for details.(note - some unexplained variations in the derivation of 'access tile' have been observed. 33 1 y coordinate of 'access tile'
34 2 Walker 1 ID. Refers to entry in the Walker Table where the sprite data is held. This is usually the principal walker, cartpusher or market trader. Zero if walker not issued 
36 2 Walker 2 ID. As above - usually this is the labor recruiter or market buyer

40 2? Not found this used
42 2? (not fully investigated) Number of (50-tick) spawning cycles elapsed since walker returned to building. When this number increments to a trigger level, which depends on the building type and its staffing level, a walker is spawned

For barracks, this number resets when a guard or soldier emits and increments to 9.

Initially set for all buildings (whether they spawn walkers or not) according to the value of the 2-bit and the 4-bit in the random number underlying the building's northern tile (derived from another section of the game file). Value 00 = NE, 02 = SE, 04 = SW and 06 = NW. Determines the 'quadramble' direction which will be associated with the next walker. When a walker spawns, this value is rotated clockwise.
45 1 Housing - 01 = access to fountain water. Fountain/baths - 01 = access to full reservoir. Reservoir - 01 = reservoir full 00 = reservoir dry.

Well, that's what I make of it. Only took me about two weeks, day and night

Note - any mention of 'cycle' refers to the principal game cycle of 50 'ticks', i.e. 1/16th of a month. Now those notes referred to:

Note 1 - Offset 8 (Grid Offset). All C3 maps are set in the middle of a 162x162 grid. Thus a 160x160 map has 1 tile of 'border' all around it, while a 100x100 map has 31 tiles of border. Therefore, the northern corner of a 160x160 map (x=y=0) is offset 162 values into the grid array while that of a 100x100 map is offset 5053 values into the grid. The value at offset 2 is the map coordinate converted into a grid offset and is used to retrieve values quickly from the various grids stored elsewhere in the game file.

Note 2 - Offset 16 (Consecutive number). As you place buildings in game they are stored in the buildings table in the same order in which you build. However, 'gaps' are created when buildings are removed from the list, perhaps through deletions or housing mergers. Much like files deleted from your hard drive, the gaps left are then filled in by subsequent build operations so that the list ends up mixed up in order. The value at offset 16 stores a unique consecutive number (which starts at 00) which is not affected by changes to the list order. Since only 2 bytes are used, a city in which a great number of buildings have been deleted and rebuilt might run into a hitherto unrecognised game limit when you try and build the 65,357th.

Despite the existence of this number the game neverteless uses the order of appearance in the list for such things as iron mine collapses (always the first mine found), claypit collapses (ditto) and immigration.

Note 3 - Offset 48 (Composite Building Links) A warehouse is a composite of 9 buildings, a hippodrome is actually 3 buildings and a fort is 2. Because of the way gaps are filled in the buildings table there is no guarantee that the multiple entries for such buildings will be stored together. This offset provides a means of finding the entry for neighboring components - the previous component at offset 48 and the next at offset 50. For a warehouse, the northern tile appears first and has no value at 48. The link at offset 50 is to the bay SE of the northern tile, which links back at its offset 48 to the entry for the northern tile and forwards at offset 50 to the eastern tile. The second and third rows of bays are similarly dealt with, the southern tile having no link forward. For a hippodrome the chain is NW-Centre-SE and for a fort it's main building to parade ground.

Note 4 - Offset 66 I didn't previously know that a 2x2 building which catches fire becomes 4 'burning buildings', each with its own entry in the buildings list (a 3x3 creates 9 entries). These may fill existing holes in the list, but of course they will become holes again when they burn themselves out

Note 5 - Offset 116 (Tax accumulator) Rather than storing the actual amount of tax collected from a building, the game stores the number of people taxed multiplied by the appropriate tax multiplier for the housing level. This monthly figure is accumulated throughout the year in offset 116 for each month in which the house is registered. It appears to be used to calculate the 'tax collected so far' figure in the overlay (and in the info-click dialog) from the current tax rate. That figure will be incorrect if tax rates have changed during the year. You can see this if you pause a game, look at how much tax a house has paid so far, change the tax rate and look again. Clearly no change has been made to the tax actually collected, but the figure will nevertheless change.

effect twice a month, but suffers the three-point penalty at the month end.

This caused it to oscillate a couple of times around the 50-point boundary. It developed a crime bar (and a protestor) at month end, then lost it mid-month, then developed another (and another protestor) at the next month end.
